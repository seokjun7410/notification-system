# 요구사항 해석 및 개선 의견

## 1. 해석 방향

- 과제의 핵심은 **비동기 처리, 실패 관리, 중복 방지, 운영 복구**라고 해석했습니다.
- 따라서 필수 로직을  완성하는 데 집중했습니다.

---

## 2. 주요 요구사항 해석

## 2.1 알림 발송 요청

- “알림 발송 요청 등록”은 즉시 발송이 아니라 **발송 요청 접수**로 해석했습니다.
- 그래서 API 요청 스레드는 실제 발송을 수행하지 않고 `notification`, `notification_dispatch` 를 저장한 뒤 즉시 응답합니다.
- 실제 발송은 별도 worker가 비동기로 처리합니다.
- 요구사항의 “참조 데이터”는 이번 제출에서 `eventId` 하나를 대표 참조키로 단순화했습니다.
  - 강의, 결제, 기타 이벤트마다 별도 컬럼을 늘리기보다 공통 이벤트 키 하나로 시작하는 편이 과제 범위에 더 적절하다고 판단했습니다.
  - 확장 시에는 `referenceType`, `referenceId`, `referencePayload(JSON)` 구조로 일반화하는 방향을 염두에 두었습니다.

---

## 2.2 상태 관리

- 알림 상태는 원본 알림이 아니라 **채널별 발송 단위**에 있어야 한다고 해석했습니다.
- 같은 이벤트라도 `EMAIL` 은 실패하고 `IN_APP` 은 성공할 수 있기 때문입니다.
- 따라서 `notification` 은 알림 원본, `notification_dispatch` 는 채널별 발송 상태를 담당합니다.
---

## 2.3 재시도 정책

- 재시도 가능한 실패만 retry policy 대상이라고 해석했습니다.
- 반면 재시도해도 의미가 낮은 실패는 즉시 `FAILED` 로 종료하도록 분리했습니다.
- timeout은 자동 재시도하지 않고 `UNKNOWN` 으로 처리했습니다.
  - timeout은 실제 발송은 성공했지만 응답만 받지 못한 상황일 수 있기 때문입니다.
  - 이 경우 자동 재시도를 수행하면 중복 발송 위험이 있습니다.

---

## 2.4 중복 발송 방지

“동일 이벤트 중복 발송 금지”는 두 단계로 해석했습니다.
- 첫째, 같은 알림 원본이 중복 생성되면 안 됩니다.
```text
(eventId, recipientId, type)
````

- 둘째, 같은 알림 원본에 같은 채널 발송이 중복 생성되면 안 됩니다.

```text
(notificationId, channel)
```

- 이렇게 한 이유는 같은 이벤트라도 `EMAIL` 과 `IN_APP` 은 각각 한 번씩 발송될 수 있어야 하기 때문입니다.
- 동시 요청에 대해서는 애플리케이션 사전 조회만으로 부족하므로 DB unique 제약을 함께 사용했습니다.
- 또한 기존 `(eventId, recipientId, type)` 에 대해 다른 `title`, `content` 가 들어오면 같은 멱등 요청이 아니라 충돌 요청으로 보고 `409` 로 거절하도록 해석했습니다.

---

## 2.5 비동기 처리 구조

실제 메시지 브로커는 사용하지 않고 DB polling worker 구조로 구현했습니다.

```text
API 요청
  -> DB에 발송 대상 저장
  -> 즉시 응답

worker
  -> DB에서 발송 대상 조회
  -> 발송 처리
  -> 결과 저장
```

- worker는 `FOR UPDATE SKIP LOCKED` 를 사용해 여러 인스턴스가 같은 발송 건을 동시에 처리하지 않도록 했습니다.
- worker는 발송 대상을 한 건씩 선점하고 바로 처리합니다. 아직 발송을 시작하지 않은 row가 오래된 `PROCESSING` 으로 남는 구간을 줄이기 위한 선택입니다.
- worker는 실제 외부 호출 직전에 `sendStartedAt` 을 별도 기록합니다. recovery는 이 값을 보고 아직 외부 발송을 시작하지 못한 row와 이미 외부 호출에 들어간 row를 구분합니다.
- worker와 recovery는 서로 다른 스케줄러 풀을 사용합니다. 단일 인스턴스에서도 장시간 발송 때문에 recovery가 같이 멈추지 않도록 분리했습니다.
- 또한 `NotificationSender` 인터페이스를 두어 이후 실제 이메일 provider로 확장할 수 있도록 했습니다.

---

## 2.6 재시작 복구 가능성

- 오래된 `PROCESSING` 상태는 worker 비정상 종료 가능성이 있다고 해석했습니다.
- 서버가 재시작 되어도 복구할 수 있어야 하므로 스케줄러를 통해 복구 프로세스를 구현하였습니다.
- 다만 stale `PROCESSING` 을 모두 같은 방식으로 처리하지 않고,
  - 아직 외부 발송을 시작하지 못한 건은 `PENDING` 으로 되돌려 자동 재처리하고,
  - 이미 외부 호출을 시작한 건은 `UNKNOWN` 으로 격리해 중복 발송 위험을 피하도록 했습니다.

## 3. 개선 의견

## 3.1 구현 개선 의견
시간이 더 있다면 다음을 개선하고 싶습니다.
* Redis를 이용한 중복 검증
  -   빠르고 낮은 연산 비용으로 중복 차단

* 사용자 알림 조회 Redis 캐싱
  - 첫 페이지 조회 캐싱 등을 통해 사용 알림 조회 최적화

* 채널(Sender) 확장성 코드
  - Map 기반 Factory에 구현체 선택을 위임하여 채널 추가 시 서비스 코드 수정 없이 확장 가능하도록 OCP를 강화
 
* 가상스레드 도입
  - I/O기반 작업이 많은 알림 서버 특성상 가상스레드로 전환하면 자원 효율성 향상이 기대됨

* `UNKNOWN` 처리 고도화
  - 현재는 멱등키/상태조회 부재로 수동 처리, 향후 지원 시 자동 상태 전이 또는 재시도

* CircuitBreaker 적용
  -   외부 발송 채널 API 장애 시 불필요한 재시도 방지

* 실패/격리 상태 운영 기능
  -   `FAILED`, `UNKNOWN` 조회 및 수동 재시도 API

* DB 아카이브
  -   오래된 알림 데이터 이관으로 성능 유지
